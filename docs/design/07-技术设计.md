## 5. 核心算法

### 5.1 合理史低计算（IQR 方法，修改）

```
输入：某品类/子类+匹配条件下的所有历史报价 prices[]
步骤：
  1. Q1 = percentile(prices, 25)
  2. Q3 = percentile(prices, 75)
  3. IQR = Q3 - Q1
  4. lower = Q1 - 1.5 × IQR
  5. upper = Q3 + 1.5 × IQR
  6. filtered = prices[lower <= p <= upper]
  7. reasonable_low = min(filtered)         ← 合理史低（主基准）
  8. baseline_avg = mean(filtered)          ← 历史均价（辅助）
  9. baseline_median = median(filtered)     ← 中位价（辅助）
  10. historical_min = min(prices)           ← 绝对最低（仅提醒）
输出：reasonable_low, avg, median, std, cv, p10, p90, min, max
```

### 5.2 偏差率与色标判定（修改）

```
deviation_pct = (new_price - reasonable_low) / reasonable_low

色标判定（阈值可按品类配置）：
  |deviation| ≤ yellow_threshold  → "normal"（无色）
  yellow_threshold < |deviation| ≤ red_threshold → "yellow"（需关注）
  |deviation| > red_threshold → "red"（异常）

默认阈值：yellow = 5%, red = 10%
```

不再使用 green/blue 报警体系，不再使用修正 Z 分数。

### 5.3 横向对比矩阵算法（新增）

```
输入：project_id, category, supplier_ids[]
步骤：
  1. 从 quotes 表拉取该项目+品类下所有报价
  2. 优先使用“用户确认过的对齐方案”生成比价行；没有对齐方案时按 material_id 生成初始矩阵
  3. 对每个比价行：
     a. 按匹配条件（品类扩展属性中 role="匹配" 的字段）查询历史数据
     b. 计算 reasonable_low（含对应项目和日期）
     c. 计算各供应商 deviation_pct = (price - reasonable_low) / reasonable_low
     d. 标记 is_lowest、alert_level
     e. 推荐 = 偏差最小的供应商
  4. 底部汇总：各供应商已报材料总价、平均偏差
  5. 推荐主供 = 总评分最高（价格+完整度+品牌合规综合）
```

### 5.4 AI 报价对齐复核算法（设计中）

#### 背景

同一标的多份投标文件通常来自同一份招标清单，但供应商报价 PDF 的表达并不一致：

- 同一清单项可能被识别为不同名称：如 `Y型过滤器`、`不锈钢Y型过滤器`、`给排水 Y型过滤器`。
- 同一清单项的型号可能被拆到不同字段：如 `DN20` 与 `DN20 PN16YZ`。
- 模型可能把“合价”识别成“单价”，导致金额异常偏高。
- 仅靠固定词表或硬编码规则会覆盖不全，也容易误合并不同清单项。

因此该能力不直接自动改库，而是作为“AI 建议 + 用户确认”的复核流程。

#### 触发条件

比价流程中满足任一条件时触发 AI 复核：

1. 同一项目下供应商数量 ≥ 2，且矩阵中存在大量“未报价”单元格。
2. 多个物料名称/规格相似，但分散在不同 material_id 上。
3. `unit_price × quantity` 与 `total_price` 偏差超过阈值（默认 1%），疑似列错位。
4. 某供应商报价完整度明显低于其他供应商，且原始文件行数相近。

#### 输入数据

`POST /api/analysis/bid-alignment/suggest`

```json
{
  "project_id": 59,
  "category": "阀门",
  "supplier_ids": [7, 58, 59],
  "rows": [
    {
      "quote_id": 11986,
      "supplier_name": "上海泰科龙",
      "material_name": "Y型过滤器",
      "spec": "DN20",
      "unit": "个",
      "quantity": 1,
      "unit_price": 69.12,
      "total_price": 69.12,
      "source_file": "泰科龙投标文件.pdf",
      "source_page": 1,
      "source_row": 3
    }
  ]
}
```

数据来源以 `quotes + materials + suppliers + extraction_jobs.result` 为主；后续应补充 source metadata（文件名、页码、行号、原始 OCR 文本片段），便于用户核对。

#### 大模型任务

模型只输出“建议”，不得直接写库：

1. 识别疑似同一招标清单项的报价组。
2. 给每组建议一个标准行名、标准规格、单位、数量。
3. 标出每家供应商对应的 quote_id。
4. 标出字段纠错建议，例如“unit_price 可能应为 total_price / quantity”。
5. 给出置信度和原因，原因必须引用可核对字段（名称、规格、数量、单位、价格、相邻行顺序）。

输出示例：

#### 用户确认与落库

前端展示“待确认对齐建议”面板：

- 左侧显示原始识别行：供应商、文件、页码、原名称、规格、数量、单价、合价。
- 右侧显示 AI 建议：标准行名、标准规格、合并组、字段纠错。
- 用户可逐组操作：确认、拒绝、拆分、手动调整。
- 用户确认后才调用 `POST /api/analysis/bid-alignment/apply`。

后端落库建议：

- 新增 `bid_alignment_groups`：存储一次比价任务中用户确认的标准行。
- 新增 `bid_alignment_items`：存储 quote_id 到 alignment_group_id 的映射，以及字段纠错覆盖值。
- 不直接覆盖 `materials` 和 `quotes` 原始识别值，保留可追溯性。
- `bid_matrix` 优先按 alignment_group_id 聚合；未确认的行仍按 material_id 展示。

#### 约束

- 置信度低于阈值（建议 0.75）不得默认勾选。
- 数量、单位明显冲突时必须提示用户，不得自动合并。
- 材质、型号、品牌等差异作为“差异说明”保留，不作为默认阻断项；是否阻断由品类属性配置决定。
- 所有 AI 建议必须可撤销，且保留操作日志。

### 5.5 供应商评分

```
total = Σ(dimension_score × weight)

dimensions:
  price:        基于平均偏差率（与合理史低比）线性映射到 [20, 100]
  history:      win_count >= 5 → 100, >= 3 → 80, >= 1 → 60, 0 → 40
  completeness: valid_quotes / total_quotes × 100
  brand:        品牌档位命中率 × 100
  commercial:   人工录入（默认 60）
```

### 5.6 匹配条件分组

比价时按品类扩展属性中 `role = "匹配"` 的字段分组，只有匹配字段全部相同的报价才放在一起比较。`role = "差异"` 的字段允许不同，在报告中以备注形式说明。

### 5.7 配电箱特殊处理

配电箱不按整箱比价，按元器件级别拆分：
- 导入时解析 BOM 结构，每个元器件生成独立 material + quote 记录
- 匹配条件：元器件名称 + 规格 + 品牌系列
- 差异字段：品牌具体型号

### 6.2 认证流程

```
1. 路由守卫检查 localStorage 中的 mempas_token
```
